MCBB 3.0 is Fiserv’s next-generation cloud networking product. Designed to work 
seamlessly with the new Global Backbone, MCBB 3.0 is set to replace all existing cloud 
networking solutions currently in use at Fiserv.
The product is engineered to be scalable, easy to deploy and operate, and rich in 
features. To formalize its capabilities, a baseline requirements list has been compiled 
and made available for vendor review.
While it’s understood that not every design requirement may be fully met, vendors are 
strongly encouraged to propose alternative solutions to address any gaps. These 
proposals may include leveraging third-party products owned or certified by Fiserv, as 
well as utilizing internal Fiserv capabilities such as automation.
The scope of the solution  as a minimum has to cover the following CSP regions:
eu-west-1 (Ireland)
eu-central-1 (Frankfurt)
ap-southeast-1 (Singapore)
ap-southeast-2 (Sydney)
ap-south-1 (Mumbai)
ap-south-2 (Hyderabad)
us-west-2 (Oregon)
us-east-1 (N. Virginia)
us-east-2 (Ohio)
me-central-1 (UAE)
sa-east-1 (Sao Paulo)
The geo regions not covered in the list, but desired to be in scope of the solution:
Argentina
Panama
Mexico
Fiserv owns multiple Cloud OnRamp circuits across various regions. The solution must 
support short-path routing from each region to the nearest domestic on-premises WAN 
or, if unavailable, to the closest regional on-ramp. The Cloud OnRamp WAN may consist 
of dedicated circuits ranging from 10 to 100 Gbps, or hosted WAN capacity provisioned 
through existing or new carriers. Traffic within AWS should be routed internally within 
the cloud service provider (CSP).

Page 9All network traffic must be encrypted.  That requirement includes either CSP backbone or 
Cloud OnRamp of any scale
The solution must either provide, or clearly indicate the availability of, low-latency 
s within the same region. connectivity options with other CSP
The Fiserv Cloud network comprises multiple routing domains with overlapping address 
spaces. The proposed MCBB3.0 solution must support 1:1 network address translation 
to enable communication between overlapping segments.
Route propagation should be granular down to specific prefixes, with configurable 
options to allow, deny, or automatically aggregate routes to broader subnets where 
applicable. All routes must be tagged with Fiserv-required BGP communities.  Routing 
tables across the CSP network must be fully visible and accessible via API, CLI, or UI. All 
routing policies must be explicitly controlled by Fiserv, and the CSP must not inject 
unsolicited routes into the routing tables.
The proposed solution must offer a consistent feature set across all regions. Any 
limitations in features or scalability must be clearly defined and aligned with Fiserv-
specific requirements.
The solution must not include hidden components, such as inline load balancers, that 
could potentially impact Fiserv traffic.It is highly desirable if the new solution offers a 
level of bandwidth protections against applications with bulk consumption. 
Network telemetry is expected to provide Fiserv with real-time visibility into traffic flows, 
top talkers, fat flows, and related metrics. Additionally, network-related data should be 
retrievable from the CSP infrastructure to support internal capacity planning and alerting.

Cloud WAN based design
Overview
AWS Cloud WAN is preferred solution to establish connectivity to/between/from Fiserv 
sites and AWS regions. There is a number of use cases to be supported, explored, tested, 
and documented in the MCBB 3.0 design.
The following baseline assumptions must be accounted for:
The US environment is large-scale, utilizing 10–100GE on either BB or Cloud 
OnRamp.
The US environment supports colocation with NaaS providers such as Megaport or 
Equinix.
The US Cloud OnRamp is preferably based on dedicated Direct Connect links.
The US BB is a multisite network connecting data centers and edge locations.
The US Cloud OnRamp is preferably deployed at edge locations.
The international environment is small to medium scale, typically using sub-GE, 
1GE, or 10GE.
A standard international deployment consists of two data centers within the same 
country, either in metro or national proximity.
International environments generally do not have a BB; instead, they rely on DCI 
links of sub-10GE or 10GE+.
International sites are typically not colocated with AWS Edge sites in the same data 
center facility.
Third-party WAN/NaaS (if available) must be used for most international sites to 
establish Direct Connect links with AWS.

Cloud WAN with Split DXGW Design
The Cloud WAN (C-WAN) segment is extended across all AWS regions, with local VPCs 
attached to Cloud Network Edges (CNEs). Each region includes a set of domestic Direct 
Connect (DX) connections delivered to various DX edge sites.
AWS supports affinity between DX locations and AWS regions, as detailed at https://aws.
. By default, an AWS region learns on-premises amazon.com/directconnect/locations/
routes from AWS DX through its associated regional DX location.

--- Page 16 ---
Page 16If a regional DX location is unavailable or fails, AWS Direct Connect Gateway (DXGW) can 
redirect traffic to other DX connections attached to the same DXGW. When multiple 
alternative DX connections are available, traffic is distributed across them using Equal-
Cost Multi-Path (ECMP) routing. This behavior necessitates a design decision to limit the 
scope of DXGW to regional boundaries. Doing so prevents scenarios where AWS regions 
might attempt to route traffic through DX connections located in different geopolitical 
areas. AWS solution addressing these scenarios is to use dedciated DXGW per region
Figure 2. Baseline network diagram split DXGW
The C-WAN segment does not propagate local VPC routes to remote DXGW attachments. 
Instead, route propagation is limited to CNE-to-CNE communication and to the DXGW 
associated with the local CNE. Consequently, the DXGW serves both as a route 
aggregator and as the data plane between different C-WAN regions when forwarding 
traffic to on-premises locations. From a technical perspective, all routes ingested by CNEs 
into the DXGW are advertised to every DX connection attached to that DXGW.
