❶ Ingress (Internet → Workload EC2)

Internet → Regional IGW (per-region, e.g. IGW use1) → Workload VPC Public Subnet → ALB receives traffic → WAF inspects L7 → ALB forwards to GWLBe (in workload VPC) → GWLBe routes via ingress routing table to C-WAN Attachment → C-WAN Prod Segment → C-WAN policy: send-to Inspection Segment → Inspection VPC C-WAN Attachment Subnet → GWLB (GENEVE encapsulation) → Palo Alto NVA inspects (IPS/IDS, threat prevention, URL filtering) → returns to GWLB → C-WAN Attachment Subnet → re-enters C-WAN Prod Segment → back to Workload VPC → EC2 in private subnet
Key: Even though each workload VPC has its own IGW and ALB/WAF (decentralized ingress), Cloud WAN is in the path. All traffic is centrally inspected via GWLB + Palo Alto before reaching EC2. The GWLBe in the workload VPC is what redirects ingress traffic into the C-WAN inspection path.

Egress (Workload EC2 → Internet)

EC2 (private subnet) → Workload VPC route table: 0.0.0.0/0 → cwan-att → C-WAN Attachment in workload VPC → C-WAN Prod Segment → C-WAN policy: send-to Inspection Segment → Inspection VPC C-WAN Attachment Subnet → GWLB (GENEVE) → Palo Alto NVA inspects outbound (URL filtering, data exfiltration prevention) → returns to GWLB → Inspection VPC FW Subnet route table: 0.0.0.0/0 → Regional NAT GW → Regional NAT GW (e.g. Regional USE1 NGW) → NAT translates private → public IP → Regional IGW → Internet
Key: Egress does NOT use the workload VPC's NAT GW. Traffic exits through the Regional NAT GW associated with the Inspection VPC in the Network/Security Account. This ensures all outbound traffic is inspected by Palo Alto before leaving AWS. Each region has its own Regional NGW for egress.

①-⑤ East-West (Prod VPC ↔ Non-Prod VPC)

① EC2 in Prod VPC → VPC route table: 0.0.0.0/0 → cwan-att → traffic enters C-WAN Prod Segment →
② C-WAN segment policy: send-to Inspection Segment (Prod cannot reach Non-Prod directly) →
③ Inspection VPC C-WAN Attachment Subnet → GWLB (GENEVE) → Palo Alto NVA inspects (microsegmentation, inter-segment policy) →
④ returns to GWLB → C-WAN Attachment Subnet → re-enters C-WAN → routed to Non-Prod Segment →
⑤ traffic delivered to Non-Prod VPC → EC2 in Non-Prod
Key: Segment isolation is enforced by C-WAN policy. Prod and Non-Prod are separate segments — they cannot communicate without traversing the Inspection segment. This is the "send-to" routing mode. Same pattern applies for any cross-segment communication.

North-South Outbound (Workload EC2 → On-Prem DC)

EC2 (Prod VPC) → VPC route table: on-prem prefixes (e.g. 10.0.0.0/8 → cwan-att) → C-WAN Prod Segment → C-WAN policy: send-via Inspection Segment → Inspection VPC C-WAN Attachment Subnet → GWLB (GENEVE) → Palo Alto NVA inspects → returns to GWLB → C-WAN Attachment Subnet → re-enters C-WAN → Hybrid Segment → DXGW for region (e.g. dxgw-use1) → Dual DX connections (redundant, per edge connect) → On-Prem S-DMZ → Fabric / CloudOne (Fiserv Global Backbone) → destination Data Center (USOMA01, USCHD01, UCHCZ01, DEFRA, APAC)
Key: The "send-via" policy means traffic must pass through Inspection before reaching the Hybrid segment. The DXGW is geo-bound per region — dxgw-use1 serves us-east-1, dxgw-use2 serves us-east-2, dxgw-usw2 serves us-west-2. No cross-geo leakage. Each DXGW has dual DX connections for redundancy at the edge connect location. Traffic traverses the Fabric/CloudOne backbone to reach the target DC.

North-South Inbound (On-Prem DC → Workload EC2)

On-Prem Data Center (e.g. USOMA01) → Fabric / CloudOne backbone → On-Prem S-DMZ → Dual DX connections (redundant via Megaport / edge connect) → DXGW for target region (e.g. dxgw-use1) → C-WAN Hybrid Segment → C-WAN policy: send-via Inspection Segment → Inspection VPC C-WAN Attachment Subnet → GWLB (GENEVE) → Palo Alto NVA inspects inbound on-prem traffic → returns to GWLB → C-WAN Attachment Subnet → re-enters C-WAN → Prod Segment → Prod VPC → EC2
Key: Same inspection enforcement in reverse. On-prem traffic entering via the Hybrid segment MUST traverse Inspection before reaching any workload segment. The split DXGW design ensures traffic stays geo-local — US DCs use US DXGWs. Traffic originates from a DC, traverses the Fabric/CloudOne backbone, passes through the S-DMZ, and enters AWS via redundant DX connections.

M&A Flow (Acquisitions Account → Workload EC2)

Acquisition workload → Existing FiservMnA TGW → Transit Gateway Attachment Route Table (CloudWAN) → TGW peering attachment → C-WAN MnA Segment → C-WAN policy routes to Inspection Segment → Inspection VPC C-WAN Attachment Subnet → GWLB (GENEVE) → Palo Alto NVA inspects (M&A traffic is untrusted) → returns to GWLB → C-WAN Attachment Subnet → re-enters C-WAN → Prod Segment → Prod VPC → EC2
Key: M&A traffic enters Cloud WAN via the legacy TGW peering attachment into a dedicated MnA segment. This keeps acquired company traffic isolated in its own segment and always inspected before reaching production workloads. Two FiservMnA TGWs provide redundancy for the M&A integration path.

Shared Services (Any Segment → Shared Svc VPC)

EC2 in any workload VPC → C-WAN workload segment (Prod or Non-Prod) → C-WAN segment peering (Prod ↔ Shared Svc, Non-Prod ↔ Shared Svc) → Shared Services Segment → Shared Services VPC → R53 Resolver Endpoints, CloudWatch, S3 Log Buckets
Key: Shared Services peers directly with Prod and Non-Prod segments via C-WAN segment peering. This is the only cross-segment path that does NOT require Inspection traversal — because Shared Services only hosts infrastructure services (DNS, logging, monitoring), not application workloads.

Design Principles Summary

Principle Implementation
Decentralized Ingress Each workload VPC has its own IGW + ALB/WAF + GWLBe. Per-region IGW provides the internet entry point.
Decentralized Egress Each region has its own Regional NAT GW (in the Inspection VPC / Network Account). Egress exits through the regional NGW after inspection.
Centralized Inspection ALL traffic (ingress, egress, E-W, N-S, M&A) must traverse the Inspection VPC where GWLB distributes to Palo Alto NVA targets via GENEVE encapsulation. Only exception: Shared Services peering.
Cloud WAN in ALL Paths C-WAN segment policy (send-to / send-via) enforces that no traffic bypasses inspection. Segment isolation prevents direct cross-segment communication.
DXGW per Region Each region has its own DXGW (dxgw-use1, dxgw-use2, dxgw-usw2) inside the Hybrid segment. No cross-geo leakage.
Dual DX per Edge Connect Each DXGW has dual Direct Connect connections for redundancy at each edge connect location.
Fabric / CloudOne On-prem traffic traverses the Fiserv Global Backbone / IBB SD-WAN overlay to reach the target Data Center (USOMA01, USCHD01, UCHCZ01, DEFRA, APAC).
M&A Isolation Acquired companies enter via dedicated MnA segment through legacy FiservMnA TGW peering. Always inspected before reaching production.
