1. AWS Cloud WAN Terminology & Concepts

Quick reference for key AWS Cloud WAN terms and policy constructs used throughout this lab build.

1.1 Core Components

TermDefinition
Core NetworkThe global C-WAN resource that spans all regions; contains the policy document, segments, and attachments.
Core Network Edge (CNE)A regional presence of the core network. Each AWS region in your policy gets a CNE that handles routing and forwarding for that region.
SegmentAn isolated routing domain within C-WAN, similar to a VRF. Attachments in the same segment can communicate; different segments cannot unless explicitly peered.
Segment PeeringAllows route exchange between two segments within the same CNE. Used to enable Shared Services to reach both Prod and Non-Prod.
AttachmentA connection between a resource (VPC, DXGW, S2S VPN, TGW, or Connect) and the core network. Each attachment belongs to exactly one segment.
Attachment PolicyRules that automatically map new attachments to segments based on tag conditions. Evaluated in order; first match wins.
Network RouteA route entry in a segment’s route table at a specific CNE, showing destination prefix, attachment, and route type (propagated or static).

1.2 Policy Actions & Routing Constructs

TermDefinition
Segment ActionsPolicy rules that define how traffic is routed between segments and attachments. Includes share-with, send-to, and send-via actions.
Send ToForwards traffic destined for a specific segment to an intermediary attachment (e.g., Inspection VPC) before delivery. Used for single-hop inspection.
Send ViaInserts a middlebox (e.g., firewall) into the traffic path between two segments. Supports both single-hop and dual-hop inspection patterns.
Single HopTraffic passes through one intermediary attachment before reaching destination. Example: Prod VPC → Inspection VPC → On-Prem via DXGW.
Dual HopTraffic passes through two intermediary attachments before reaching destination. Example: Source VPC → Inspection (region A) → Inspection (region B) → Dest VPC.
Share WithShares routes from one segment with another without full peering. Provides one-directional or selective route sharing.
When (Condition)A condition block in the attachment policy that matches on resource tags (e.g., tag:Environment = Prod) to determine segment placement.
Edge OverrideAllows per-region overrides of the global segment/attachment policy. Used when a specific CNE needs different routing behavior.

2. LAB Overview & Objectives

2.1 Purpose

The MCBB 3.0 AWS Lab environment serves as a pre-production validation platform to prove out the Cloud WAN with Split DXGW architecture defined in the MCBB 3.0 production design document (v.25, Jan 2026). The lab mirrors the production topology at a reduced scale, enabling the team to validate design requirements (DRs), test failure scenarios, refine Terraform modules, and document operational runbooks before production deployment.

2.2 Scope

  • AWS Cloud WAN (C-WAN) core network across three lab regions: us-east-1, ap-southeast-2, eu-central-1
  • C-WAN segmentation: Prod, Non-Prod, Shared Services (peered to Prod + Non-Prod), and IBB segments
  • Cloud Network Edge (CNE) attachments per region
  • Split DXGW architecture with regional DXGW instances
  • Multiple VPC types per region: Prod, Non-Prod, Inspection, and Shared Services
  • Inspection VPC for firewall deployment (AWS Network Firewall or third-party appliance)
  • Direct Connect integration via Megaport virtual cross-connects (VXCs)
  • BGP route propagation validation with community tagging
  • Cross-region traffic flow validation over C-WAN backbone
  • International Backbone (IBB) overlay design — cloud-native approach replacing Versa SD-WAN
  • Service chaining through inspection VPCs for east-west and north-south traffic
  • Infrastructure as Code (Terraform Enterprise) with GitLab CI/CD runners
  • Network observability: interim CloudWatch/VPC Flow Logs, target state ThousandEyes + Splunk + Selector AI

2.3 Out of Scope

  • Full-scale 100GE Cloud OnRamp capacity testing
  • LATAM region deployment (Argentina, Panama, Mexico)
  • Inter-CSP connectivity (Azure, GCP integration points)
  • QoS enforcement at egress (DR 4.5 — not supported by CSP)
  • Versa SD-WAN / SDR overlay — removed from design due to operational issues

2.4 Key Design Change: Versa Removal

The original MCBB 3.0 design specified Versa SDRs for the International Backbone (IBB) SD-WAN overlay. Due to recent operational issues with Versa, the IBB overlay will be re-architected using cloud-native alternatives:

  • AWS Cloud WAN native cross-region routing with dedicated IBB segment and route policies
  • AWS Site-to-Site VPN with accelerated VPN (Global Accelerator backed) for encrypted cross-region tunnels
  • AWS Transit Gateway inter-region peering as a fallback for IBB traffic segregation
  • Public VIF over existing DX connections for IBB underlay (as referenced in production design doc)

2.5 Inspection VPC Architecture

Each region includes a dedicated Inspection VPC for centralized security appliances supporting east-west inspection (inter-VPC traffic routed through firewall before reaching destination), north-south inspection (traffic to on-prem/internet inspected before leaving the region), and service chaining via C-WAN routing policies (send-to / send-via actions).

Firewall options under evaluation:

  • AWS Network Firewall — cloud-native, Terraform-friendly, integrates with VPC route tables
  • Third-party appliance firewall (e.g., Palo Alto, CheckPoint, Fortinet) — deployed as EC2-based NVAs

5. BGP Community Tagging & Route Propagation

5.1 BGP Community Tagging Plan

All routes must be tagged with Fiserv-required BGP communities per DR 2.5:

Community ValueMeaningApplied To
[TBP]US origin prefixAll us-east-1 VPC routes
[TBP]APAC origin prefixAll ap-southeast-2 VPC routes
[TBP]EMEA origin prefixAll eu-central-1 VPC routes
[TBP]On-prem DC prefixRoutes from DX connections

5.2 Route Propagation Rules

  • C-WAN Prod segment: Prod VPC CIDRs propagate CNE-to-CNE within Prod segment only (isolated from Non-Prod)
  • C-WAN Non-Prod segment: Non-Prod VPC CIDRs propagate within Non-Prod segment only (isolated from Prod)
  • C-WAN Shared Services segment: Peers to both Prod and Non-Prod segments; Shared Services VPC routes visible in both
  • Inspection VPC attachment: Receives return routes from Prod/Non-Prod VPCs; sends default or specific routes to attract inspection traffic
  • dxgw-us: Receives aggregated US VPC routes from CNE-US-East; advertises to DX peers
  • dxgw-apac: Receives Sydney VPC routes from CNE-APAC-Sydney; advertises to DX peers
  • dxgw-emea: Receives Frankfurt VPC routes from CNE-EMEA-Frankfurt; advertises to DX peers
  • Cross-DXGW traffic: Must traverse C-WAN backbone (not leak between DXGWs)

7. Phased Deployment Runbook

The lab buildout is structured in sequential phases with dependencies and acceptance criteria mapped to Design Requirements (DRs).

7.1 Phase 1: C-WAN Core Network & Segments

Dependencies: AWS account provisioned, Terraform Enterprise workspace configured, GitLab runner operational
DRs: DR 4.1, DR 2.10

  • Deploy C-WAN core network with global policy document
  • Create Prod, Non-Prod, Shared Services (peered to Prod + Non-Prod), and IBB segments across all three regions
  • Deploy CNEs: CNE-US-East, CNE-APAC-Sydney, CNE-EMEA-Frankfurt
  • Validate segment health and CNE state via console/API

Acceptance: All CNEs Available; four segments visible in all regions; Terraform state clean.

7.2 Phase 2: VPC Deployment & Intra-Region Routing

Dependencies: Phase 1 complete
DRs: DR 1.1, DR 2.1, DR 2.10

  • us-east-1: Deploy Prod (10.20.0.0/21), Non-Prod (10.21.0.0/21), Inspection (10.22.0.0/22), Shared Services (TBD)
  • ap-southeast-2: Deploy Prod (10.30.0.0/21), Inspection (10.32.0.0/22)
  • eu-central-1: Deploy Prod (TBD), Non-Prod (TBD), Inspection (TBD)
  • Attach Prod VPCs to Prod segment, Non-Prod VPCs to Non-Prod segment, Shared Services to Shared Services segment
  • Configure Shared Services segment peering to both Prod and Non-Prod segments
  • Attach Inspection VPCs as separate C-WAN attachments
  • Deploy EC2 test instances in each VPC for connectivity validation
  • Validate segment isolation: Prod cannot reach Non-Prod directly
  • Validate Shared Services reachable from both Prod and Non-Prod via peering

Acceptance: Routes in correct segments; Prod/Non-Prod isolated; Shared Services peered to both; Inspection attachments visible.

7.3 Phase 3: Cross-Region C-WAN Routing

Dependencies: Phase 2 complete
DRs: DR 1.2, DR 2.2, DR 2.9

  • Validate C-WAN propagates VPC routes between CNEs across all three regions
  • Test cross-region connectivity: us-east-1 ↔ ap-southeast-2, us-east-1 ↔ eu-central-1, ap-southeast-2 ↔ eu-central-1
  • Verify route tables via API/CLI (aws networkmanager get-network-routes)
  • Measure cross-region latency baselines
  • Confirm VPC routes do NOT leak to DXGW attachments

Acceptance: Full mesh ping across all three regions; route tables correct; no DXGW leakage.

7.4 Phase 4: Split DXGW & Direct Connect Integration

Dependencies: Phase 3 complete, Megaport circuit delivered (target: March 20)
DRs: DR 1.3, DR 1.4, DR 2.3, DR 1.9, DR 1.10, DR 2.5

  • Create three regional DXGWs: dxgw-us, dxgw-apac, dxgw-emea
  • Attach DXGWs to their respective CNEs
  • Provision Megaport VXCs as hosted DX connections
  • Establish BGP peering over transit VIFs
  • Validate geo-affinity and DXGW failover
  • Confirm cross-DXGW isolation (DR 1.12)
  • Test BGP community tagging on advertised routes (DR 2.5)

Acceptance: On-prem reachable via correct regional DXGW; no cross-geo DX leakage.

7.5 Phase 5: Inspection VPC & Service Chaining

Dependencies: Phase 2 complete (can parallel Phases 3-4)
DRs: DR 1.6, DR 4.8, DR 4.9

  • Deploy firewall in each region’s Inspection VPC (start with AWS Network Firewall)
  • Configure C-WAN send-to/send-via actions to steer east-west and north-south traffic through Inspection VPC
  • Validate firewall logs show inspected flows
  • Test traffic blackhole when firewall removed (no bypass path)
  • Optionally test third-party NVA (CheckPoint, Palo Alto) as alternative in one region

Acceptance: All east-west/north-south traffic through Inspection VPC; firewall logs confirm; no bypass.

7.6 Phase 6: IBB Replacement Architecture (Post-Versa)

Dependencies: Phase 4 complete
DRs: DR 1.5, DR 1.11, DR 1.12

  • Deploy IBB C-WAN segment with dedicated route policies
  • Test candidate IBB approaches:
    • Option A: C-WAN IBB segment with route policies restricting international traffic to approved paths
    • Option B: Accelerated Site-to-Site VPN tunnels for encrypted cross-region IBB underlay
    • Option C: Public VIF on existing DX for IBB overlay
  • Validate international traffic paths and geopolitical routing constraints
  • Document recommendation with pros/cons matrix

Acceptance: At least one IBB approach validated end-to-end with encryption; geopolitical constraints enforced.

7.7 Phase 7: Observability & Telemetry

Dependencies: Phase 3+ (can parallel Phases 4-6)
DRs: DR 3.1, DR 3.3, DR 3.4

Target State (Cloud-Agnostic):

  • ThousandEyes — Network path visualization, latency monitoring across all CSPs
  • Splunk — Centralized log aggregation for flow logs, firewall logs, C-WAN events
  • Selector AI — AIOps anomaly detection and automated root cause analysis

Interim Lab (AWS Native):

  • CloudWatch metrics for C-WAN (bytes in/out, packet counts per attachment)
  • VPC Flow Logs for all lab VPCs (S3 + CloudWatch Logs)
  • Network Manager route analytics and topology visualization
  • CloudWatch alarms for DX connection state changes
  • AWS Network Firewall logs (alert + flow) from Inspection VPCs
  • Document telemetry gaps and map to target state tool capabilities

Acceptance: Interim dashboards operational; gaps documented with target state mapping.

7.8 Phase 8: End-to-End Validation & Stress Testing

Dependencies: All previous phases complete
DRs: DR 2.7, DR 2.8, DR 2.10

  • Full traffic flow matrix: every region to every region, every region to on-prem, all through Inspection VPCs
  • Blackout simulation: kill primary DX, measure convergence
  • Brownout simulation: inject latency/packet loss, observe routing behavior
  • Multi-tenancy validation: secondary segment isolation
  • iperf bandwidth tests across C-WAN backbone and DX connections
  • Compile final test results against DR matrix

Acceptance: All DRs validated or documented with gaps/workarounds; final report compiled.
