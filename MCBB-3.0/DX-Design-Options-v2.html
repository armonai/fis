MCBB 3.0 Cloud WAN — Direct Connect Architecture Design

1. On-Premises Topology

3 Data Centers connected via on-prem WAN WAN backbone. 6 DX Edge PoPs provide geographic diversity. Each PoP is an interconnect point — not a destination. Any PoP reaches any DC via on-prem WAN. All DCs advertise 10.0.0.0/8 summary.

2. Model 1: Full Criss-Cross

Max 4 Regions Fastest Failover High Complexity

Every DX PoP has Transit VIFs into all 3 DXGWs. Every DXGW has Transit VIFs into all 3 CNEs. This is the design from the original architecture image.

Failover Tiers

TierTriggerPathLatency Impact
1SJC DX circuit failsDXGW-West still has Denver VIF → traffic exits DenverNone
2All West DX PoPs fail (SJC+Denver)DXGW-Central has VIF already at US-West-2 CNE → exits Chi/DalNone (no backbone hop)
3West + Central DX failDXGW-East VIF at US-West-2 CNE → exits ATL/AshNone (no backbone hop)

Traffic Engineering

  • Layer 1 — Within DXGW (SJC/Denver pair): No preference — ECMP across both DX PoPs. Both are equal paths to on-prem WAN. DXGW load-balances. If one PoP fails, the other takes 100%. DC-level routing is on-prem WAN's job.
  • Layer 2 — Local DXGW preference: Cloud WAN routing policy — prepend-asn-list inbound on cross-region DXGW attachments at each CNE, so local DXGW is always preferred
  • Layer 3 — Inter-region failover order: Cloud WAN associate-routing-policy on CNE-to-CNE edge pairs — prepend on distant regions to control which cross-region DXGW is preferred when local DXGW fails
  • Note: Each DXGW has 3 Cloud WAN attachments (one per CNE) — routing policies can be applied per attachment for granular control
⚠️ VIF Limit: Each DX circuit uses 3 VIFs (one per DXGW). AWS limit is 4 VIFs per DX connection. At 4 regions = 4 DXGWs = 4 VIFs per circuit (at limit). At 5 regions this design breaks.

3. Model 2: Regional DXGW — Local VIF Only

Unlimited Regions Low Complexity Lowest Cost

1 DXGW per region. Each DX PoP pair feeds into its local DXGW only (1 VIF per circuit). Cross-region failover via Cloud WAN backbone. Inter-site WAN carries rerouted traffic between DX locations.

  • Intra-DXGW (SJC vs Denver): ECMP — no preference. Both PoPs are equal paths to on-prem WAN. DXGW load-balances across both VIFs.
  • Inter-region failover order: Cloud WAN associate-routing-policy with prepend-asn-list on CNE-to-CNE edge pairs
  • Cannot apply different Cloud WAN policies per circuit (both behind same DXGW attachment)

4. Model 3: Per-Circuit DXGW with Cloud WAN Routing Policies

Unlimited Regions Full Visibility Tested & Validated ✅

1 DXGW per DX circuit (6 total). Each gets its own Cloud WAN attachment. Cloud WAN routing policies control all traffic engineering. This is what we tested.

Failover Tiers

2Both West DXGWs failBoth withdraw → Cloud WAN backbone → deterministic next region (routing policy)+60-100ms

Traffic Engineering (Fully Validated ✅)

  • Per-PoP DXGW preference: Cloud WAN routing policy — prepend-asn-list inbound on one DXGW to make the other preferred within same CNE
  • Inter-region failover order: Cloud WAN associate-routing-policy + prepend-asn-list on CNE-to-CNE pairs
  • Per-region ASNs: East=65001, West=65002, EU=65003 — enables independent TE per location
  • Per-circuit Cloud WAN visibility: Each DXGW has its own attachment — see BGP routes per circuit

5. Model 4: Hybrid — Local DXGW + Global Backup DXGW

Unlimited Local No On-Prem WAN Dependency Medium Complexity

3 local DXGWs handle normal traffic. 1 global DXGW receives a VIF from one PoP at each location — provides AWS-backbone-only backup. No on-prem WAN dependency for Tier 2.

Traffic Engineering

  • Intra-DXGW (SJC vs Denver): ECMP — no preference. Both PoPs are equal paths to on-prem WAN. If one PoP fails, the other takes 100%.
  • Local vs Global DXGW: Cloud WAN routing policy — prepend-asn-list inbound on Global DXGW attachment to make it less preferred than local DXGWs
  • Inter-region failover via Global DXGW: Global DXGW receives VIFs from multiple DX locations — Cloud WAN sees one attachment with multiple paths. On-prem BGP controls which DX location the Global DXGW prefers
  • Limitation: Global DXGW is a single Cloud WAN attachment — cannot apply per-location routing policies within it

7. Recommendation

🏆 For Scale + Full Control: Model 3 (Per-Circuit DXGW)

Best for enterprise environments needing per-circuit traffic engineering from the Cloud WAN side. Scales infinitely. Full visibility. Tested and validated — all 3 regions, all failover tiers, deterministic inter-region routing confirmed.

  • Per-region ASNs (65001/65002/65003) with Cloud WAN routing policies for per-PoP DXGW preference
  • Cloud WAN routing policies for deterministic inter-region failover order
  • Requires reliable on-prem WAN for Tier 2 failover last-mile

🥈 For Simplicity: Model 2 (Regional DXGW)

Fewest moving parts. DXGW ECMPs across both PoPs within each region. Choose this if on-prem WAN is reliable and you don't need per-circuit Cloud WAN visibility.

🥉 For No On-Prem WAN Dependency: Model 4 (Hybrid)

If on-prem WAN between DX locations is unreliable, the Global DXGW ensures AWS backbone carries all failover traffic without touching on-prem WAN.

⚠️ For 3 Regions Only: Model 1 (Full Criss-Cross)

Maximum redundancy and fastest failover (no backbone hop). But hard limit at 4 regions. Only choose if you're certain you won't scale beyond 4 regions.
